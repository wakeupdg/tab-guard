**OI-003 (child-tab URL constraint bypass) — implemented, partially browser-verified:**
- `background.js`: `restrictedChildTabs` session-storage tracking for blank-start child tabs; `HTMLFormElement.prototype.submit` hook (confirmed working — forms use real shared prototype methods, no platform restriction); submitter `formAction`/`formTarget` override handling; `document.baseURI`-based relative URL resolution.
- One real bug found and fixed: `isAmbiguousManualNewTab()` originally treated an empty/falsy tab start-URL the same as a manual Cmd+T new tab (skip tracking) — but a script-initiated blank `window.open()` can also report empty/falsy at that exact instant, not just the literal `"about:blank"` string assumed at design time. Fixed to track empty/falsy the same as `"about:blank"`, narrowing the skip case to only the two literal `chrome://newtab/`/`chrome://new-tab-page/` strings.
- A second, related bug in `main-world-blocker.js`: `window.open('about:blank')` (explicit string) was falling through to the mismatch check and getting blocked outright, unlike `window.open()` with no argument — fixed to treat both identically.
- **Both fixes are confirmed present on disk but were never cleanly re-verified against a truly fresh extension reload before the session ended** — see Next Steps.

**OI-004 (Origin Boundary Lock) — implemented, one piece confirmed non-functional:**
- `manifest.json`: `declarativeNetRequest` permission added.
- `background.js`: per-tab DNR session rules (allow-origin + block-else, `main_frame` only) for **child tabs** — this part is solid, unaffected by anything below.
- `content-bridge.js`: strips `meta[http-equiv=refresh]` tags while armed (declarative-navigation fallback).
- **`main-world-blocker.js`'s `Location.prototype.replace`/`.assign`/`href` hooks (the armed tab's own tab-under protection) do not work and never did.** Confirmed via direct property inspection: `window.location.replace`/`.assign` are **own properties of the `Location` instance**, `writable: false, configurable: false` — not inherited from `Location.prototype` at all. Patching the prototype has zero effect; the real native method is called every time regardless. This was not caught by code review or by Cory's own automated testing (which never triggers the `confirm()` dialog Scenario 4 requires, per the session's dialog-avoidance rule) — it was found because the **user manually clicked through the real dialog** and got redirected to `evil-tab-under.com` on a tab that was genuinely, confirmedly armed at the time.
